Athena — Project Research

Personal pipeline for current structured facts — stock open, today's weather, esports match times — from sites/APIs on demand or on a schedule, with a path to auto-onboard sources and an information dashboard.

Executive summary

Scope Athena as a layered ingestion pipeline, not a pure scraper. Prefer APIs and direct connectors for structured facts; use HTML heuristics when no API exists; escalate to browser/LLM only for exceptions. RSS is optional — useful when a feed already exists.

Heterogeneous data rule: unify the output contract (DataPoint), not extraction logic. Each source declares a kind + connector.

Recommended: thin custom core (registry, scheduler, SQLite) + reusable connector handlers. Defer Celery/Temporal until ~50 sources.

Path A — Adopt + customize Spike changedetection.io or jaypetez/glean. Fastest time-to-value; less dashboard control.
Path B — Build lean custom Python worker + SQLite + Next.js dashboard + YAML configs. Full control; more upfront work.

Core architecture — adapter pattern

Sites differ at fetch/extract time but converge on a small set of value shapes. One interface: fetch → extract → normalize → DataPoint[].

Layer 1 — Source config (per site)
kind stock_quote · weather · schedule · text · custom
connector api_json · html_selector · ical · rss · browser
extract jsonpath / css / ical parser / schema
▼
Layer 2 — Connector runtime (~6 handlers)
Reusable handlers; no per-site Python code
▼
Layer 3 — Normalized DataPoint (all sources)
{ source_id, kind, label, value, unit, as_of, meta }

Example configs — same framework, three domains

# Stock — API, not Yahoo HTML
id: aapl-open  kind: scalar  connector: api_json
fetch:  url: "https://www.alphavantage.co/query?function=GLOBAL_QUOTE&symbol=AAPL&apikey=${AV_KEY}"
extract: path: "$.Global Quote.05. open"  transform: [{ op: float }]
normalize: label: "AAPL open"  unit: USD
schedule: "0 9 * * 1-5"

# Weather — Open-Meteo (free, no key)
id: nyc-today  kind: forecast  connector: api_json
fetch:  url: "https://api.open-meteo.com/v1/forecast?latitude=40.71&longitude=-74.01&daily=..."
extract: map: { high: "$.temperature_2m_max[0]", rain_pct: "$.precipitation_probability_max[0]" }
schedule: "0 6 * * *"

# Esports — iCal first
id: lol-schedule  kind: schedule  connector: ical
fetch:  url: "https://example.com/schedule.ics"
extract: fields: [summary, dtstart, location, description]
schedule: "0 */4 * * *"

Findings

Unified framework — adapter + DataPoint envelope

Core answer: unify envelope and connector interface; specialize only in reusable handlers and category templates.

adapterDataPointconfig-driven

+ Advantages

  • Add site = add config, not code
  • Dashboard renders by kind
  • APIs more stable than finance HTML
  • Manual + cron share same path

− Disadvantages

  • ~6 connector types to design upfront
  • Edge cases need custom or LLM
  • API keys & rate limits per category
Relevance — Foundation for all disparate sites. Many websites → few connectors → few kinds → one dashboard.

Prefer APIs over scraping for structured facts

Prices, temperature, indices — public APIs beat brittle CSS on finance/weather UIs.

api_jsonstocksweather

+ Advantages

  • Stable JSON schemas
  • Built-in units & timestamps
  • Clearer ToS than scraping

− Disadvantages

  • Rate limits (AV free: 25 req/day)
  • Niche esports lack APIs
  • Vendor dependency
Relevance — Registry field origin: api | scrape. Wizards: stock → ticker+API; weather → lat/lon+Open-Meteo; esports → iCal/API → table scrape.

Per-source fetch: API & direct extraction first

Declare how to reach data directly — not RSS-first discovery. RSS/sitemaps only when already available.

+ Advantages

  • Matches stocks/weather/esports reality
  • Fewer parts than RSS-first
  • Clean connector / store / UI split

− Disadvantages

  • Per-source config when no API
  • Selectors break on redesign
  • JS pages need browser
Relevance — Registry: connector: api_json | html_selector | ical | rss | browser based on what the site offers.

Extraction stack — trafilatura + Playwright + LLM

Heuristics for articles; Playwright when static fetch empty; LLM for enrichment or irregular layouts only.

+ Advantages

  • trafilatura for article bodies
  • Playwright only on failure
  • LLM on new items only

− Disadvantages

  • Fails on tables/dashboards
  • Playwright ~300MB+
  • LLM needs validation
Relevance — fetch: httpx | playwright, extract: trafilatura | css | llm_schema. Start httpx only.

Declarative YAML configs (Scrapit pattern)

URL, backend, selectors, transforms, validation — one file per target; UI writes same schema later.

Relevance — Model for source registry. Version configs; audit selector changes.

Storage — SQLite first

One file, zero ops. Tables: sources, fetch_runs, items, optional snapshots. Postgres when multi-user or concurrent workers.

Relevance — Store extracted values + hashes, not raw HTML blobs. DuckDB/Postgres only at scale.

Scheduling — cron / APScheduler for v1

Daily personal use doesn't need Celery/Temporal. Isolate failures per source (try/except). Queue at ~50 sources.

Relevance — 0 6 * * * daily + athena fetch --source X. Same entrypoint for manual runs.

Dashboard — DataPoints by kind, not feed reader

Scalar cards, forecast, schedule lists, charts. Borrow Hati layout patterns; don't mimic RSS UX.

Relevance — MVP: source health table + widgets by kind + search. Phase 2: configurable layout.
Links
Hati

Compliance — robots.txt, rate limits

Honor robots.txt, 1–5s delays, descriptive User-Agent, no login walls, minimize PII. Not legal advice.

Relevance — Fetch layer checks robots.txt; per-domain delay; robots_checked_at on source. Login-required = out of scope v1.

Recommendations

  1. Spike first. Docker-run changedetection.io + jaypetez/glean with 5 real URLs. If 80% fit, extend; else build custom.
  2. Phase 1 only if building. Python + SQLite + api_json, html_selector, ical + athena fetch --source X. Add rss only when feed URL known.
  3. Registry schema now. Fields: url, connector, schedule, extract, normalize, tags, enabled, last_status. Phase 4 UI writes rows — no pipeline rewrites.

Open questions

  • Categories first? Stocks + weather (API-heavy) vs esports (more scraping) — pick 2 for v1.
  • Historical depth? Current value only vs time-series (30-day AAPL chart) — schema impact.
  • Change vs snapshot? Alert on change, or always show latest?
  • Manual add UX? Web form, YAML, CLI, or browser extension?
  • Hosting? Local Mac, home server, or VPS — affects Playwright + cron.
  • LLM enrichment? Summaries/tags — API keys + cost.
  • Multi-user? Personal only vs shared dashboard — auth + Postgres.
  • Adopt threshold? What is "good enough" after spike?
