Athena — Architecture Overview

Documentation of the initial architecture conversation: a personal dashboard for task prioritization and daily planning, with a deterministic ingestion pipeline feeding an AI planning agent.

Documented 2026-07-11 Conversation summary 2–3 sources · few-minute polling Companion to ingestion-research.html

Executive summary

Athena is a personal system that collects structured facts from a small set of sources (calendar, tasks, etc.), stores them reliably, and uses an AI agent to help prioritize work and plan each day.

Core insight: ingestion and reasoning are separate concerns. A scheduled, deterministic worker fetches and normalizes data into SQLite. The agent reads that store — it does not poll external APIs on every request.

At personal scale (2–3 sources, polling every few minutes): Python worker + SQLite + APScheduler + YAML source configs. No Celery, no Playwright, no LLM in the fetch path.

Conversation context

Starting goal

Build a dashboard that helps prioritize tasks and plan each day. The project may use AI agents, but this is a first agentic system — so the right foundation matters.

Initial assumption (revised)

An agent might handle information ingestion — fetching from calendar, task managers, and other sources.

Prior research in ingestion-research.html argued against this: ingestion should be a layered pipeline with config-driven connectors, not an agent loop. This conversation confirmed that recommendation for a planning dashboard at small scale.

Scale constraints

~2–3 external sources, refreshed at most every few minutes. Personal, single-user. APIs preferred over scraping where available.

Key decision: agent as planner, not fetcher

Ingestion layer — deterministic Scheduled worker polls APIs on a cron. Fetch → extract → normalize → upsert into SQLite. No LLM. Failures isolated per source. Same code path for manual athena fetch and scheduled runs.
Agent layer — reasoning Reads normalized tasks, events, and user goals from SQLite. Produces a ranked daily plan. Runs on demand (user opens dashboard, clicks "reprioritize") — not on every poll cycle.
Why not agent-driven ingestion? Unpredictable latency, token cost on every poll, hard-to-test behavior, and opaque failures. Structured API responses don't need interpretation — they need reliable extraction and storage.

Agent responsibilities: resolve calendar conflicts, weigh priority vs. deadlines, suggest time blocks, explain tradeoffs. Not responsibilities: HTTP requests, JSON parsing, cron scheduling.

Dashboard

Reads SQLite directly for items and plans (fast, no agent latency on every page load). Widgets by kind: task lists, schedule, stat cards. Source health table shows last fetch status. "Reprioritize" button triggers agent asynchronously.

# Worker pseudocode
for source in registry.enabled():
    if source.is_due():
        try:
            raw = connectors[source.connector].fetch(source)
            items = connectors[source.connector].extract(raw, source)
            store.upsert(items)
        except Exception as e:
            store.log_failure(source.id, e)

Agent-driven fetch vs. deterministic pipeline

− Agent-driven ingestion

  • Unpredictable latency per poll
  • Token cost on every refresh
  • Hard to test and replay
  • Opaque failure modes
  • Re-fetches on every agent turn

+ Deterministic pipeline

  • Sub-second per API call
  • Free after setup (no LLM in loop)
  • Same input → same output
  • Clear per-source logs
  • Agent reads cache, not live APIs

Polling best practices

  • Use updated_since / sync tokens where APIs support them — pull deltas, not full lists.
  • Store content_hash or external_updated_at to skip no-op writes.
  • Per-source intervals: not everything needs the same cadence.
  • Webhooks (Todoist, Linear) can replace polling later; start with polling for simplicity.
